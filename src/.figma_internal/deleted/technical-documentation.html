1. Application Overview

Executive Summary

Artistrial is a comprehensive web application designed for the artist and media industry, providing a unified platform where artists, studios, investors, and industry professionals can connect, collaborate, and conduct business. The platform serves as a multi-faceted marketplace supporting various aspects of the creative industry from asset trading to talent acquisition, legal services, and educational resources.

Mission Statement

To create a centralized ecosystem that empowers artists and creative professionals by providing the tools, connections, and resources needed to succeed in the modern digital creative economy.

Key Objectives

  • Facilitate seamless transactions between artists and buyers across multiple marketplaces
  • Enable collaborative project management and team formation
  • Provide comprehensive business management tools for creative professionals
  • Support lead generation and customer relationship management
  • Create a unified user experience across all platform modules

Target Users

Artists & Creators

Individual artists, designers, and content creators looking to showcase and sell their work

Studios & Agencies

Creative studios and agencies seeking talent, resources, and collaboration opportunities

Investors

Individuals and organizations looking to invest in creative projects and talent

Service Providers

Legal professionals, educators, and service providers supporting the creative industry

Architecture Patterns

  • Three-Tier Architecture: Clear separation between presentation, application, and data layers
  • Component-Based Design: Modular React components for reusability and maintainability
  • Route-Based Code Splitting: Optimized loading through lazy-loaded components
  • State Management: Context API for global state and local state for component-specific data
  • RESTful API Design: Standardized API endpoints following REST conventions

Data Flow

  1. User Interaction: User interacts with React components in the frontend
  2. State Updates: Component state or global context is updated
  3. API Calls: Frontend makes HTTP requests to Supabase Edge Functions
  4. Authentication: Requests are authenticated using Supabase Auth
  5. Data Processing: Edge functions process requests and interact with database
  6. Response: Data is returned to frontend and UI is updated

Key-Value Store

The application utilizes a flexible key-value store for rapid prototyping and dynamic data:

Table: kv_store_f6985a91 Columns: - key (VARCHAR) - Unique identifier - value (JSONB) - Flexible JSON data storage - created_at (TIMESTAMP) - updated_at (TIMESTAMP)

KV Store Operations

  • get(key): Retrieve single value
  • set(key, value): Store single value
  • mget(keys): Retrieve multiple values
  • mset(data): Store multiple key-value pairs
  • del(key): Delete single key
  • mdel(keys): Delete multiple keys
  • getByPrefix(prefix): Retrieve keys by prefix pattern

10. API Documentation

API Architecture

The application uses Supabase Edge Functions with Hono web framework for API endpoints:

Base Configuration

Base URL: https://{projectId}.supabase.co/functions/v1/make-server-f6985a91/ Headers: - Authorization: Bearer {publicAnonKey} - Content-Type: application/json

Authentication Endpoints

User Registration

POST /signup Body: { email: string, password: string, user_metadata: { name: string, is_artist?: boolean } } Response: { user: User, session: Session }

User Authentication

POST /auth/signin Body: { email: string, password: string } Response: { user: User, session: Session, access_token: string }

Cart Management

Add to Cart

POST /cart/{userId}/add Body: { artworkId: string } Response: { cart: { items: CartItem[], total: number } }

Get Cart

GET /cart/{userId} Response: { items: CartItem[], total: number }

Remove from Cart

DELETE /cart/{userId}/remove/{itemId} Response: { cart: { items: CartItem[], total: number } }

Data Management

Key-Value Store Operations

// Get single value GET /kv/{key} Response: any // Set single value POST /kv/{key} Body: any Response: void // Get multiple values POST /kv/mget Body: { keys: string[] } Response: any[] // Set multiple values POST /kv/mset Body: { [key: string]: any } Response: void // Get by prefix GET /kv/prefix/{prefix} Response: any[]

File Upload

Upload File

POST /upload Content-Type: multipart/form-data Body: FormData with file Response: { url: string, path: string, signedUrl?: string }

Error Handling

Standardized error response format:

Error Response: { error: string, message: string, status: number, details?: any } Common Status Codes: - 200: Success - 400: Bad Request - 401: Unauthorized - 403: Forbidden - 404: Not Found - 500: Internal Server Error

Rate Limiting

  • Authentication Endpoints: 10 requests per minute
  • Data Endpoints: 100 requests per minute
  • File Upload: 20 uploads per hour
  • Search Endpoints: 50 requests per minute
